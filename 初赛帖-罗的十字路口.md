## 2. Demo 创作思路

### 灵感来源

我是罗永浩的狂热粉丝。从新东方讲课的段子、牛博网、砸西门子冰箱、做锤子手机、到脱口秀大会、直播带货"真还传"——一路追随至今。不是追星那种追随，是**想从罗老师的一言一行里，获取人生智慧**的那种追随。

追了这么多年，有一个观察在我心里憋了很久：

> **如果用利益最大化来衡量，老罗的很多选择都是"错"的。**
>
> 砸冰箱——商业上愚蠢，但定义了"较真"。
> 做锤子手机——烧光所有钱，但"天生骄傲"不是空话。
> 拒绝烂广告——少赚几千万，但保住了信任。
> 直播还债——放下身段，但"真还传"反而赢得更多尊重。
>
> **正是这些"错"的选择，让他成为了老罗。**

每次看他站在十字路口做选择，我都会想：**如果是我，我会怎么选？我会不会有勇气砸那台冰箱？会不会放下身段去直播还债？** 现实里的人生没法存档读档，但这个"如果"的念头一直在我心里。直到我开始用 TRAE，才终于能把这份多年的体悟做成一款游戏——不是评判他选对选错，而是让每个玩家都能亲自站在那些十字路口，做出自己的选择。

这让我意识到一个被大多数人生模拟器忽略的事：**现实里的人生，从来不是"选对加分、选错扣分"。** 每个选择都在定义"你是什么样的人"，但没有任何一个选择能定义你的全部。

### 想解决的问题

市面上的"选择类"游戏有三个普遍痛点：

1. **纯文字 + emoji 太单调**——缺乏视觉沉浸感，玩家代入不进去
2. **重度养成一局太久**——学习成本高，碎片时间玩不动
3. **给玩家"标准答案"**——选 A 得好结局，选 B 得坏结局，本质上在评判玩家

我想做的不是一个"游戏"，而是一份**生动的、可玩的老罗自传**——把生活道理、人生选择、和一点诙谐幽默揉在一起。让玩家在 10 分钟里，亲历老罗人生中那些关键十字路口，在笑声里咂摸出一点生活道理，然后被温柔地告知：**你这个结局，值得尊重。**

### 为什么做这个方向

- **IP 自带流量与话题性**：老罗的每次选择都上热搜，玩家天然有代入动机
- **情绪价值稀缺**：当下年轻人普遍焦虑，需要一个低成本的方式审视自己的选择逻辑
- **传播钩子明确**：通关后可生成竖版分享卡（六维雷达图 + 人生关键词），天然适合朋友圈/抖音/小红书传播
- **技术与创意可兼得**：像素风 + Canvas 2D 渲染对 AI 协作友好，TRAE 能深度参与从剧情到代码的全流程

### 作品定位：一份诙谐的、可玩的罗永浩自传

我追了老罗这么多年，最大的感受是：**老罗的人生本身就是一部最好的"人生选择教科书"。** 但教科书没人愿意读，自传又太长。所以我换了个形式——做成像素风游戏。

- **生活道理**：每个剧情节点背后都有真实的历史背景注释，玩家在玩的过程中，会不知不觉吸收老罗的人生智慧——比如"较真"的代价、"妥协"的代价、"放下身段"的代价
- **诙谐幽默**：老罗本来就是段子手出身，游戏里的对白保留了他的语言风格——"我这个人就是嘴欠"、"天生骄傲"、砸冰箱时的较真劲——让沉重的人生选择不至于太压抑
- **真实洒脱**：游戏不评判任何选择，35 种结局每个都值得尊重。我想传递的是——**老罗最打动人的从来不是他成功了，而是他活得真实。** 不管是砸冰箱的较真、做手机的执拗、还是直播还债的放下身段，他始终在选"我想成为什么样的人"，而不是"什么选择最划算"。

**这份自传想告诉当代年轻人的，其实就一句话**：

> **在你人生的十字路口，别问哪个选择最划算，问自己——哪个选择能让你活得真实洒脱。**
>
> 老罗的很多选择从利益看是"错"的，但他活成了老罗。
> 你不必成为老罗，但你可以像他一样，选属于你自己的路。

### 设计哲学（贯穿全项目的灵魂）

> **选择没有对错，只有不同的人生。**
>
> 1. 不评判选择——没有标准答案，每条路都有风景
> 2. 选择塑造身份——属性不是分数，而是"你是什么样的人"的映射
> 3. 代价即意义——放弃的东西定义了你坚持的东西
> 4. 所有结局都值得尊重——"传奇"不比"幸存者"更好，只是不同
> 5. 重玩不是为了赢，而是为了体验另一种人生
> 6. **活得真实，比活得"正确"更重要**——这是老罗教我的事

## 5. 开发心得与踩坑复盘（额外加分项）

### 5.1 沉淀的 11 条硬工程约束

5 轮 PDCA 闭环跑下来，我和 TRAE 一起沉淀了 11 条项目级硬约束，写进了项目 memory，防止问题回归：

1. `localStorage` key for play count 必须统一为 `luohammer_play_count`，确保 `replay_bonus` 天赋生效
2. 选项按钮必须有 `_choiceLock` 机制，防止快速点击或快捷键重复触发
3. 移动端布局必须包含 `env(safe-area-inset-*)`，防止刘海设备遮挡内容
4. 资源加入项目前必须转为 WebP 格式，优化加载速度
5. 所有 fixed 定位 UI 必须包含 `env(safe-area-inset-*)`
6. 定时器必须作为实例属性存储，场景 shutdown 时清理，防止内存泄漏
7. DOM 元素与状态层级应缓存，消除每帧 DOM 查询
8. 按钮交互必须包含防抖机制，防止重复触发
9. `AudioContext` 应跨场景共享，`destroy()` 时不关闭
10. 存档加载必须包含类型校验与版本迁移 hooks
11. 屏幕方向判定必须使用 `matchMedia` API，适应分屏和折叠屏

### 5.2 三个印象最深的踩坑

**坑 1：跨周目存档 key 不一致**
- 症状：`replay_bonus` 天赋在第二周目失效
- 排查：TRAE 全项目搜索发现 `luohammer_play_count` 有两种变体
- 修复：统一 key + 写入硬约束防止回归
- 心得：**AI 协作的最大价值不是写新代码，而是快速定位跨文件的一致性问题**

**坑 2：BGM 内存泄漏与后台失步**
- 症状：切场景几次后浏览器卡顿，后台标签页切回来 BGM 节奏错乱
- 排查：oscillator 引用未清理 + setTimeout 调度与实际播放脱钩
- 修复：改用 Web Audio API 精确时间调度，清理 oscillator 引用
- 心得：**音频是 AI 协作的盲区，需要把 Web Audio API 的调度模型讲清楚给 TRAE**

**坑 3：TTS 朗读与剧情推进不同步**
- 症状：玩家反馈"内容切换太快，语音还没读完就进入下一段"
- 排查：原实现用固定延时，未等待 TTS end 事件
- 修复：监听 `onend` 事件 + `_ttsResumeTimer` 防 Chrome 长文本截断
- 心得：**用户体验问题往往不是 bug，而是同步逻辑设计缺陷**

### 5.3 方法论心得：Loop Engineering 与 TRAE 智能体的契合

PDCA 闭环之所以能在 5 轮内完成 25 项修复，关键在于 TRAE 智能体的"规划-执行-验证"能力与 PDCA 的 Plan-Do-Check-Act 天然契合：

- **Plan**：我把问题清单交给 TRAE，它帮我排优先级、拆任务
- **Do**：TRAE 智能体逐项执行，并行调度多个 worker 处理独立任务
- **Check**：跑 92 步 QA 走查，TRAE 帮我验证修复无回归
- **Act**：把修复经验沉淀为硬约束，写入项目 memory

**最大的心得**：AI 协作不是"让 AI 帮你写代码"，而是"让 AI 帮你跑闭环"。前者产出代码，后者产出工程能力。

## 6. 后续优化方向与复赛展望 ⭐

先说实话：**这个初级版本我自己知道离"好"还有多远。** 作为第一次做游戏的人，我能把它跑到能上线、能通关、零报错，已经是 TRAE 帮我撑出来的极限了。但下面这几个问题，我一直记在心里，是接下来想攻克的：

### 6.1 我自己清楚知道的短板

| 维度 | 现状 | 问题 | 想做的优化 |
|------|------|------|-----------|
| **声音** | 程序化 Web Audio API 合成 BGM + TTS 语音朗读 | 程序合成的 BGM 还是偏"电子琴味"，缺情感层次；TTS 语音音色单一，沉浸感不足 | 接入更专业的音频方案（见下方复赛展望） |
| **UI** | 暗金像素美学已成型，CRT 扫描线 + 像素符号 | 部分界面排版仍显粗糙，结局分享卡的视觉冲击力还不够；移动端竖屏布局有优化空间 | 引入更精细的视觉资源 + 重做分享卡 + 移动端竖屏适配 |
| **剧情** | 70+ 节点 + 35 结局 + 100+ 随机事件 | 部分节点的文案还偏粗，情感张力没拉满；老罗人生中还有一些关键十字路口没覆盖到（比如老罗讲段子那段其实是个隐喻）；诙谐感和生活道理的平衡还能更精准 | 扩充剧情节点 + 文案润色 + 补全更多真实历史十字路口 + 让"自传感"更完整 |
| **代入感** | 8 种角色姿态 × 12 种场景 | 角色表情还不够丰富，关键节点的情绪传递偏弱 | 增加角色表情帧 + 关键节点特写 + 心跳节奏视觉反馈强化 |
