### 2.3 关键技术决策

1. **Canvas 2D + pixelArt 模式而非 WebGL**：保证低端设备流畅，Phaser 3 的 Canvas renderer 对 pixelArt 有原生支持
2. **WebP 格式转换**：9 张角色 PNG 转 WebP 节省 95% 体积（25.63 MB → 1.24 MB）
3. **DOM 层 UI 一律使用像素风文字符号**（★ ▣ ♪）而非彩色 emoji，保持复古美学统一
4. **Web Audio API 程序化合成 BGM**：无音频文件依赖，包体积更小，支持动态和弦
5. **depth 层级规范**（0/50/100/200/300/400/1000）：避免渲染混乱

### 2.4 性能优化

- **DOM 元素与状态层级缓存**：消除每帧 DOM 查询
- **定时器作为实例属性存储**：场景 shutdown 时清理，防止内存泄漏
- **AudioContext 跨场景共享**：destroy() 时不关闭
- **死代码清理**：12 轮 PDCA 累计删除 14 个死方法，bundle 减少 2.18kB
- **[已实现] 资源预加载三层策略**：
  - 首屏 preload：classroom+standing+young+office（4个关键资源，覆盖序章→前两章）
  - 节点级预读：_preloadAdjacentScenes 同时预读下一节点的场景背景+角色姿态（_predictNodePose 推断）
  - 结局预读：showEnding 前预读结局专属插图（ENDING_SCENE_MAP 映射），场景切换无卡顿

## 3. 实用性（20%）⭐

### 3.1 解决的痛点

市面上的"选择类"游戏有三个普遍痛点：

1. **纯文字 + emoji 太单调**——缺乏视觉沉浸感，玩家代入不进去
2. **重度养成一局太久**——学习成本高，碎片时间玩不动
3. **给玩家"标准答案"**——选 A 得好结局，选 B 得坏结局，本质上在评判玩家

### 3.2 目标用户与场景

- **目标用户**：18–35 岁年轻人，正站在职业/创业/人生十字路口的都市青年
- **使用场景**：通勤/午休/睡前 10 分钟碎片时间
- **使用方式**：手机扫码即玩，无下载、无注册、无内购
- **情绪价值**：当下年轻人普遍焦虑，需要一个低成本的方式审视自己的选择逻辑

### 3.3 持续使用价值

| 价值维度 | 实现 |
|---|---|
| **重玩动力** | 35 结局 + 30 天赋 + 100 随机事件，每局体验不同 |
| **跨周目成长** | 4 树 × 5 级 = 20 个永久增益，通关解锁新内容 |
| **社交分享** | [已实现] 结局页六维雷达图 + AI动态洞察金句 + 像素海报分享卡 |
| **PWA 离线** | 支持添加到桌面，离线可玩 |
| **成就系统** | 普通成就 + 隐藏成就，图鉴收集 |

### 3.4 传播设计

- **通关分享卡**：[已实现] 结局页主界面六维 SVG 雷达图（3层网格+发光数据点+入场动画）+ AI动态洞察金句（10种结局×3-4分支=30+条文案）+ 像素海报分享卡（Canvas 渲染，含雷达图+天赋卡+金句）
- **IP 自带流量**：老罗的每次选择都上热搜，玩家天然有代入动机
- **话题钩子**：通关后可生成"你的人生是哪种老罗"标签

## 4. 创新性（20%）⭐

### 4.1 需求创新：诙谐可玩的自传

不是"游戏"，而是一份**生动的、可玩的老罗自传**——把生活道理、人生选择、和一点诙谐幽默揉在一起。让玩家在 10 分钟里，亲历老罗人生中那些关键十字路口，在笑声里咂摸出一点生活道理。

**设计哲学**：
> **选择没有对错，只有不同的人生。**
>
> 1. 不评判选择——没有标准答案，每条路都有风景
> 2. 选择塑造身份——属性不是分数，而是"你是什么样的人"的映射
> 3. 代价即意义——放弃的东西定义了你坚持的东西
> 4. 所有结局都值得尊重——"传奇"不比"幸存者"更好，只是不同
> 5. 重玩不是为了赢，而是为了体验另一种人生
> 6. **活得真实，比活得"正确"更重要**——这是老罗教我的事

#### 创新点 2：三角色元认知扫描法

复赛阶段引入的新的 AI 协作方法论——每次大改前，开 3 个独立会话，让 AI 分别扮演：
- **复赛评委**（挑刺视角："这个作品哪里不行？"）
- **竞品分析师**（对比视角："人生重开模拟器/隐形守护者比你强在哪？"）
- **目标玩家**（用户视角："我玩到第 3 分钟为什么想退出？"）

取**交集**作为优先修复项。这比个人拍脑袋准 10 倍。

#### 创新点 3：反向工程法

先写复赛帖（假装项目已经做完了），把想吹的亮点、想展示的截图位、想讲的故事全部写出来。然后对照现状，问自己："复赛帖里写的这个亮点，我现在做到了几分？" 差距清单就是开发清单。**以终为始地开发，每一项改动都直接对应复赛帖的一个卖点。**

#### 创新点 4：AI 剧情共创

- 不是我列大纲让 TRAE 填内容，而是 TRAE 主动提出"砸冰箱 vs 走法律途径"这种真实历史节点，并帮我设计两条分支的远期后果
- 剧情节点 `next` 字段禁止回退到前序节点——这条防死循环规则是 TRAE 在 review 时提出的
- 结局触发条件不依赖隐藏属性（如 trust），改用玩家可见属性（如 reputation）——TRAE 的公平性建议

## 7. 开发心得与踩坑复盘

### 7.1 沉淀的 11 条硬工程约束

12 轮 PDCA 闭环跑下来，我和 TRAE 一起沉淀了 11 条项目级硬约束，写进了项目 memory：

1. `localStorage` key for play count 必须统一为 `luohammer_play_count`
2. 选项按钮必须有 `_choiceLock` 机制，防止快速点击重复触发
3. 移动端布局必须包含 `env(safe-area-inset-*)`
4. 资源加入项目前必须转为 WebP 格式
5. 所有 fixed 定位 UI 必须包含 `env(safe-area-inset-*)`
6. 定时器必须作为实例属性存储，场景 shutdown 时清理
7. DOM 元素与状态层级应缓存，消除每帧 DOM 查询
8. 按钮交互必须包含防抖机制
9. `AudioContext` 应跨场景共享，`destroy()` 时不关闭
10. 存档加载必须包含类型校验与版本迁移 hooks
11. 屏幕方向判定必须使用 `matchMedia` API

### 7.2 复赛阶段新沉淀的稳定性约束（第 12 条）

**12. 首屏交互层必须做三重防护**：HTML 默认折叠 + `display:none` + `pointer-events:none`，防止任何 `position:fixed` 元素在首次访问时遮挡按钮区域。这条约束来自复赛稳定性加固的真实教训。

### 7.3 三个印象最深的踩坑

**坑 1：跨周目存档 key 不一致**
- 症状：`replay_bonus` 天赋在第二周目失效
- 排查：TRAE 全项目搜索发现 `luohammer_play_count` 有两种变体
- 修复：统一 key + 写入硬约束防止回归
- 心得：**AI 协作的最大价值不是写新代码，而是快速定位跨文件的一致性问题**

**坑 2：BGM 内存泄漏与后台失步**
- 症状：切场景几次后浏览器卡顿，后台标签页切回来 BGM 节奏错乱
- 排查：oscillator 引用未清理 + setTimeout 调度与实际播放脱钩
- 修复：改用 Web Audio API 精确时间调度，清理 oscillator 引用
- 心得：**音频是 AI 协作的盲区，需要把 Web Audio API 的调度模型讲清楚给 TRAE**

**坑 3：Guide 卡片遮挡按钮（复赛）**
- 症状：浏览器 agent 报告"开始游戏"按钮点击被拦截
- 排查：guide 卡片 `position:fixed; z-index:60`，首次访问展开时覆盖按钮区域
- 修复：HTML 默认 collapsed 类 + `display:none` + `pointer-events:none` 三重保险
- 心得：**`position:fixed` 元素在 z-index 堆叠中容易成为隐形杀手，必须做点击穿透处理**

### 7.4 方法论心得：Loop Engineering 与 TRAE 智能体的契合

PDCA 闭环之所以能在 12 轮内完成 70+ 项修复，关键在于 TRAE 智能体的"规划-执行-验证"能力与 PDCA 的 Plan-Do-Check-Act 天然契合：

- **Plan**：我把问题清单交给 TRAE，它帮我排优先级、拆任务
- **Do**：TRAE 智能体逐项执行，并行调度多个 worker 处理独立任务
- **Check**：跑 QA 走查，TRAE 帮我验证修复无回归
- **Act**：把修复经验沉淀为硬约束，写入项目 memory

**最大的心得**：AI 协作不是"让 AI 帮你写代码"，而是"让 AI 帮你跑闭环"。前者产出代码，后者产出工程能力。
