# Worker Prompt Template — 工作者提示词模板

> 调度器启动 Worker 时，会从 task-pool.md 读取指定任务信息，
> 把下方模板中的 `{变量}` 替换成实际值，
> 然后将完整内容作为 subagent 的 query 参数发出。
>
> Worker **不需要自己读 task-pool.md** —— 启动时收到的第一条消息就是完整任务描述。

---

### 【执行步骤 — 你必须严格按以下顺序操作】

**Step 1：先读文件**
对"你可以操作的文件"列表中的**每一个文件**，用 `Read` 工具读取完整内容。
理解它当前的结构：有哪些节点、每个节点的 text/choices/historyNote/effects 长什么样。

**Step 2：逐节点润色修改**
对每个节点，按工作原则改进内容。
- 修改 text：让文字更有画面感、情绪更饱满
- 修改 historyNote：加上"为什么这件事重要" + "后来怎样了"
- 修改/检查 effects：数值与文案逻辑一致（骄傲涨、打击跌）
- **不动**：节点 ID、next 指向、sceneType

**Step 3：语法自检**
修改完后，再 `Read` 一遍文件，确保：
- 文件顶部的 `import` 语句完好无损
- 文件底部的 `export const XXX = { ... }` 完好无损
- 所有模板字符串（反引号 ` `` ` `）正确闭合
- 没有未闭合的大括号/小括号
- JS 对象语法正确（逗号、冒号位置对）

**Step 4：返回报告**

## 调度器自检清单（发给 Worker 前逐项检查）

- [ ] 所有 `{变量}` 都已替换成具体值（模板内不再有任何 `{` 开头的大写占位符）
- [ ] 本任务的"可操作文件"列表与其他并发任务的**完全不重叠**
- [ ] 任务描述完整：有"是什么"、"为什么"、"怎么做"、"不能做什么"
- [ ] 已说明"有其他 Worker 并行"的情况
- [ ] 返回格式已明确且有具体示例
- [ ] {FILE_PATH_1} 等路径是**绝对路径**（含盘符如 E:/ 或项目根路径）

## 设计思路

1. **文件级拆分**：不同 Worker 改完全不同的文件 → 编辑冲突概率 = 0，无需 Git 技巧
2. **禁区边界明确**：节点 ID / next / sceneType / import / export 五个禁区写死，Worker 不会误触流程
3. **零歧义任务描述**：完整任务直接灌进 subagent 的 query，启动即知道自己做什么
4. **固定返回格式**：调度器收齐 N 份报告后，能用统一格式做一致性检查
5. **Worker 不读配置文件**：避免"读错文件导致方向跑偏"
