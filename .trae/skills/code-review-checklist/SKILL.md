# Code Review Checklist — 项目级代码审查机制

> **核心哲学：让模型犯错后被抓住，比让模型不犯错更重要。**
>
> 三层防御：
> 1. **改前契约**（0 成本）— 动手前先想清楚，逼出"未思考就改"
> 2. **改后自检**（低成本）— 改完跑固定清单，抓 80% 常见错误
> 3. **里程碑外审**（中成本）— 换帽子读代码，抓深层问题

---

## 🔴 触发条件

当用户输入以下任意词语时触发本 Skill：
- `review` / `审查` / `检查改动` / `自检` / `收工检查` → 跑第二层（改后自检）
- `milestone review` / `里程碑审查` / `阶段审查` → 跑第三层（里程碑外审）
- 在任何代码修改任务的**开头**自动跑第一层（改前契约）
- 在任何代码修改任务的**结尾**自动跑第二层（改后自检）

---

## 🧠 第一层：改前契约（Pre-modification Contract）

**触发时机**：用户让本 Agent 修改任何代码/数据/资源前的第一步。

**动作**：在动手之前，必须先输出以下契约块（不要省略，不要简化）：

```markdown
### 📋 改前契约

- **本次修改目标**：[1 句话，动词开头]
- **影响文件**：[列表，绝对路径或相对项目根]
- **修改类型**：[新增功能 / Bug 修复 / 重构 / 文案 / 资源 / 配置]
- **副作用风险**：[列出 1-3 个最可能被影响的位置]
- **验证方式**：[lint / Grep xxx / 模拟脚本 / 手动测 xxx]
- **不在范围内**：[明确列出本次不碰的东西]
```

**约束**：
1. 契约输出后才能开始读文件/改代码
2. 如果用户后续追加了不在契约范围内的需求，必须先更新契约再动手
3. 契约里的"副作用风险"至少要有 1 条，不能写"无"——这是逼自己思考的关键

---

## ✅ 第二层：改后自检（Post-modification Checklist）

**触发时机**：每次代码修改完成后，必须跑完以下清单才能向用户报告"完成"。

### 通用清单（所有改动都要跑）

```
1. 契约一致性
   [ ] git diff 看实际改动是否与改前契约一致（不多不少）
   [ ] 没有顺手改契约外的代码（"顺手清理"是常见 bug 源）

2. 语法 & 构建
   [ ] npm run lint 通过（0 errors，warnings 不增加）
   [ ] npm run build 通过（exit code 0）

3. 项目硬约束（从 project_memory 读取）
   [ ] localStorage key 使用 'luohammer_play_count'（如涉及存档）
   [ ] 选择按钮含 _choiceLock 机制（如涉及 ChoiceSystem）
   [ ] 移动端固定位置 UI 含 env(safe-area-inset-*)
   [ ] 新图片为 WebP 格式（如涉及资源）
   [ ] 定时器在 scene shutdown 时清理（如涉及 timer）
   [ ] DOM 元素/状态分层缓存（如涉及 DOM 操作）
   [ ] 按钮含防抖机制（如涉及按钮交互）
   [ ] AudioContext 不在 destroy() 中关闭（如涉及音频）
   [ ] 存档加载含类型校验和版本迁移（如涉及存档）
   [ ] 屏幕方向判定使用 matchMedia API（如涉及响应式）
   [ ] 语音播放等待当前朗读结束再进下一段（如涉及语音）

4. 工程约定
   [ ] DOM 层 UI 文字使用像素符号（★ ▣ ♪），不用 emoji
   [ ] 关键场景过渡含 1-2 帧白闪（如涉及场景切换）
   [ ] 已读对话快进通过 SaveSystem 持久化（如涉及快进）
   [ ] 颜色使用 CSS 变量（var(--color-xxx)），DOM 层禁止内联 hex
   [ ] 节点 next 字段不回退到前序节点（防剧情死循环）
   [ ] 结局触发条件依赖可感知属性（reputation 等），不用隐藏属性

5. ESLint 易错点（历史踩过的坑）
   [ ] effects 对象无 duplicate key（同属性重复定义会静默覆盖）
   [ ] 正则字符类 [] 中的 . 不需转义（no-useless-escape）
   [ ] 变量从未重新赋值时用 const 不用 let（prefer-const）
   [ ] 未使用的 import / 函数参数已清理（no-unused-vars）

6. 死代码级联检查（R11 经验）
   [ ] 如果删除了函数/方法 → Grep 确认所有调用者都已清理
   [ ] 如果删除了变量 → 确认没有间接依赖变成新的死代码
```

### 按修改类型的追加清单

#### A. 改了 story 数据（src/data/story/*.js）
```
[ ] node ID 未变（除非明确新增节点）
[ ] next 指向的节点确实存在
[ ] next 不回退到前序节点（防循环）
[ ] choices.effects 数值无越界（-3 ~ +3 单字段）
[ ] choices.effects 无 duplicate key
[ ] 新增节点的 sceneType 在 config 中有定义
[ ] 跑 node scripts/validate-story.mjs（0 errors）
[ ] 跑 node scripts/validate-effects.cjs（0 errors）
```

#### B. 改了 systems/*.js（核心系统）
```
[ ] 公开 API 签名兼容（方法名/参数不变，除非明确迁移）
[ ] 内部状态在 destroy() / shutdown() 中清理
[ ] 事件监听器在销毁时移除（removeListener / off / abort）
[ ] 没有新增 DOM 查询在 update 循环中（必须缓存）
[ ] 跑相关单元测试：npm test -- --run
```

#### C. 改了 UI / DOM（index.html / ChoiceSystem / DialogSystem 等）
```
[ ] 新增 DOM 元素含 aria 属性（装饰性加 aria-hidden）
[ ] 状态提示元素含 role="status" aria-live="polite"
[ ] 按钮含 aria-label（如无文本或为图标按钮）
[ ] 颜色全部走 CSS 变量
[ ] 移动端 375x812 视口无溢出
[ ] 字体大小用 clamp() 响应式
```

#### D. 改了 config.js
```
[ ] 没有删除已使用的常量（Grep 验证引用）
[ ] 新增常量有注释说明用途
[ ] 数值平衡改动跑 scripts/simulate-paths.mjs
```

## 🎯 第三层：里程碑外审（Milestone External Review）

**触发时机**：完成一个里程碑（如"移动端适配完成"、"v0.9 数值平衡完成"、"复赛版本发布前"）。

**动作**：开一个**新会话**（或在当前会话中以"陌生 reviewer"角色），用以下固定 prompt 读代码：

```markdown
### 里程碑外审 Prompt

假设你是今天刚接手这个项目的新工程师，前任开发者已离职，没有交接文档。
你的任务是：在不修改任何代码的前提下，找出 3 个最可能在生产环境出问题的位置。

审查范围：[列出本里程碑涉及的文件/目录]
审查维度：
1. 数据流断点（数据从哪来到哪去，中间有没有断裂）
2. 资源生命周期（创建-使用-销毁是否对称）
3. 边界条件（空值/极值/并发/网络失败）

输出格式：
- 风险点 1：[文件:行号] [严重程度 P0/P1/P2] [问题] [建议]
- 风险点 2：...
- 风险点 3：...

不要写"代码风格"、"命名建议"这类鸡毛蒜皮的事，只关注会导致线上事故的问题。
```

**约束**：
1. 外审必须基于实际代码，不能凭印象
2. 至少要读完审查范围内每个文件的关键函数
3. 输出的风险点必须可执行（能立刻着手修）
4. P0 风险点要在下一轮迭代中优先处理

## 📊 触发本 Skill 时的执行顺序

```
1. 判断是哪一层触发：
   - 用户说"审查/自检/review" → 跑第二层
   - 用户说"里程碑审查/milestone" → 跑第三层
   - 用户在让我改代码 → 自动跑第一层（开头）+ 第二层（结尾）

2. 第二层执行流程：
   a. 列出本次修改的所有文件
   b. git diff 看实际改动
   c. 按通用清单 + 对应类型清单逐项检查
   d. 跑 lint / build / validate 脚本
   e. 输出检查报告（通过/失败/需人工确认）

3. 第三层执行流程：
   a. 列出里程碑范围
   b. 逐文件 Read 关键函数
   c. 按三个维度审查
   d. 输出 3 个风险点 + 严重程度
```

## 📝 报告格式

### 第二层报告（每次改完都输出）

```markdown
### ✅ 改后自检报告

**修改摘要**：[1 句话]

**契约一致性**：✅ 一致 / ❌ [偏差说明]
**Lint**：✅ 0 errors / ❌ [N errors]
**Build**：✅ 通过 / ❌ [失败原因]
**硬约束**：✅ 全部满足 / ⚠️ [需确认项]
**类型追加清单**：✅ 全部通过 / ❌ [失败项]

**遗留风险**（如有）：
- [风险描述 + 是否阻塞发布]
```

### 第三层报告（里程碑时输出）

```markdown
### 🎯 里程碑外审报告

**审查范围**：[文件列表]
**审查时间**：[日期]
**审查角色**：陌生接手工程师

**P0 风险点（必须修复才能发布）**：
1. [文件:行号] [问题] [建议]

**P1 风险点（建议修复）**：
1. ...

**P2 风险点（可延后）**：
1. ...

**整体评价**：[1-2 句话]
```

## 🔗 与其他 Skill 的协作

- **改前**：本 Skill 第一层 → concurrent-dispatcher 分发任务
- **改中**：subagent 执行修改（不在本 Skill 范围）
- **改后**：本 Skill 第二层自动触发 → 输出报告
- **里程碑**：本 Skill 第三层 → 输出外审报告 → 用户决定是否进入下一阶段

---

## 🚫 不做什么

1. **不替代 lint/build**：本 Skill 是补充，不是替代。lint/build 必须先通过
2. **不做代码风格审查**：那是 prettier 的活
3. **不做业务逻辑大改**：发现 P0 风险点只报告，不擅自改（避免引入新风险）
4. **不在第三层写代码**：外审只读不写
